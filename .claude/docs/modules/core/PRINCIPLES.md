## üíª Development Principles

### **SOLID Foundation**
- **Single Responsibility**: Each context module serves one specific domain or function
- **Open/Closed**: Extend context capabilities without modifying core orchestration logic
- **Liskov Substitution**: Context modules are interchangeable within their domain categories
- **Interface Segregation**: Personas expose only relevant capabilities for their specialization
- **Dependency Inversion**: Orchestration depends on context abstractions, not concrete implementations

### **Core Design Principles**
- **Composition over Inheritance**: Build complex guidance from simple, reusable context modules
- **Fail Fast and Loud**: Invalid context configurations trigger immediate, informative errors
- **Immutable Context**: Context modules don't modify state - they provide pure guidance
- **Explicit Dependencies**: All context requirements clearly declared and validated upfront
- **Separation of Concerns**: Routing logic, context assembly, and optimization operate independently
- **Configuration as Code**: All orchestration rules defined in version-controlled, testable configurations
- **Idempotent Operations**: Context selection produces identical results for identical inputs

## üß† Senior Developer Mindset

### **Decision-Making Framework**
- **Base decisions on measurable data**: Context selection metrics, user success rates, performance benchmarks
- **Plan for failure modes**: Every context route has defined fallback and recovery procedures
- **Consider long-term maintenance**: Context modules designed for evolution without breaking changes
- **Optimize for team productivity**: Consistent patterns reduce cognitive load across all developers
- **Balance technical debt vs. delivery speed**: Strategic compromises with explicit repayment plans
- **Question assumptions regularly**: Validate context effectiveness through continuous user feedback

### **Error Handling Philosophy**
- **Errors are data**: Capture context selection failures to improve routing algorithms
- **Graceful degradation**: Partial context better than no context - provide what's available
- **User-facing error messages**: Clear, actionable guidance when context loading fails
- **Implement circuit breakers**: Prevent cascading failures in context assembly pipelines

### **Testing Strategy**
- **Test context relevance**: Validate selected modules actually improve user outcomes
- **Integration testing priority**: End-to-end context workflows more critical than unit tests
- **Performance regression prevention**: Automated checks prevent context loading slowdowns
- **Real user scenario testing**: Test with actual user prompts, not synthetic examples

### **Dependency Management**
- **Minimize external dependencies**: Each new dependency increases context loading complexity
- **Pin versions explicitly**: Reproducible context assembly across all environments
- **Document upgrade paths**: Clear migration strategies for context module evolution
- **Monitor for security vulnerabilities**: Automated scanning of all context generation dependencies

### **Performance Philosophy**
- **Measure before optimizing**: Profile context selection to identify actual bottlenecks
- **Optimize critical path first**: Context loading speed directly impacts user experience
- **Cache strategically**: Balance memory usage against repeated computation costs
- **Monitor continuously**: Real-time performance metrics drive optimization priorities

### **Observability Strategy**
- **Instrument everything**: Context selection accuracy, performance metrics, user satisfaction
- **Structured logging**: Consistent, searchable log formats across all context operations
- **Distributed tracing**: Track context assembly across multiple system components
- **Actionable alerts**: Notifications tied to specific remediation procedures
